# スロログ 開発ガイドライン

## プロジェクト概要
スロット収支を記録・分析するPWA（+ Capacitor iOS対応）。
React 18 + Vite + recharts + localStorage のみ。サーバーなし。

## 技術スタック
- `src/App.jsx` — メインコンポーネント（全タブのロジック・JSX）
- `src/utils/storage.js` — localStorage操作
- `src/utils/helpers.js` — 日付・表示ユーティリティ
- `src/components/ProfitStr.jsx` — 収支表示コンポーネント
- `src/styles/app.css` — 全スタイル
- `ios/` — Capacitor iOSプロジェクト（手動編集しない）

## プロダクト思想

- 技術を見せるためのアプリではなく、ユーザーの面倒を減らすためのアプリを作る
- AIや新技術を使うこと自体を目的にしない
- 「便利」「気持ちいい」「また使いたい」を優先する
- 作った本人が毎日使いたくなる品質を目指す
- 機能追加より不要な操作を減らすことを評価する
- ユーザーが考えなくていい設計を目指す

## ユーザー起点

- 作り手の自己満足ではなく、実際の困りごとを解決する
- 新機能追加より、面倒な操作をなくすことを優先する
- 「できることが増えた」より「楽になった」を評価する

## 判断基準

迷った場合は以下を優先する。

1. 毎日使いやすいか
2. 操作が増えていないか
3. 見た目が気持ちいいか
4. 長期的に愛着が持てるか

## 開発の基本方針

### シンプルを守る
- 必要最小限の変更のみ行う
- 抽象化・ライブラリ追加は指示があった場合のみ
- 指示された範囲外を勝手に変更しない

### スマホ体験最優先
- タップターゲットは 44px 以上を維持
- iOS セーフエリア（`env(safe-area-inset-*)`）を常に考慮
- アニメーション・トランジションは軽量に保つ

### 変更前に必ず行うこと
- 影響範囲をユーザーに説明してから着手する
- UIの見た目に関わる変更は事前に確認を取る
- 既存機能が壊れないことを確認してから提案する

### やってはいけないこと
- デザイン・レイアウトの判断を勝手に行わない
- 指示にないリファクタリング・整理を混ぜない
- 大規模変更を一度に行わない（小さい改善を積み重ねる）

## ビルド・デプロイ手順
```
npm run build       # Viteビルド
npx cap sync ios    # iOSへ反映
npx cap open ios    # Xcodeを開く
```

- `localStorage` キー: `surorogue_v1`（記録）、`surorogue_history`（入力履歴）
- バックアップ: JSON export/import（設定画面）
